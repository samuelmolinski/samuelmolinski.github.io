1. Azure App service

Azure app service is an inexpensive option for deploying applications (web app and APIs) at scale. It does not mean app service is not suitable for smaller apps, its just that true potential of the service is realized at scale. It has the added benefit of no maintainance overhead.

1.1.1 Accesing the deployment files

1.1.2 Directory structure of deployment

make sure to copy the jar file and the web.config file to the /site/wwwroot/ folder

Note: If the file names are incorrect you wont see any errors. Make sure the file extensions are web.config and *.jar

1.1.3 Web.config file 

<?xml version="1.0" encoding="UTF-8"?>
      <configuration>
        <system.webServer>
          <handlers>
            <add name="httpPlatformHandler" path="*" verb="*" modules="httpPlatformHandler" resourceType="Unspecified" />
          </handlers>
          <httpPlatform processPath="%JAVA_HOME%\bin\java.exe"
              arguments="-Djava.net.preferIPv4Stack=true -Dserver.port=%HTTP_PLATFORM_PORT% -jar &quot;%HOME%\site\wwwroot\gs-rest-service-0.1.0.jar&quot;">
          </httpPlatform>
        </system.webServer>
      </configuration>

The only item that needs to be configured is "gs-rest-service-0.1.0.jar"

1.2 Azure function

Azure function is the microsofts offering on serverless compute. Serverless architectures refer to applications that significantly depend on third-party services (knows as Backend as a Service or "BaaS") or on custom code that's run in ephemeral containers (Function as a Service or "FaaS"), the best known vendor host of which currently is AWS Lambda. By using these ideas, and by moving much behavior to the front end, such architectures remove the need for the traditional 'always on' server system sitting behind an application. Depending on the circumstances, such systems can significantly reduce operational cost and complexity at a cost of vendor dependencies and (at the moment) immaturity of supporting services.

2.4.2 Bridging MQTT to Azure IoT

Sample configuration file 

connection iothub-bridge1
    log_type debug
    address myiothub.azure-devices.net:8883
    remote_username myiothub.azure-devices.net/mydeviceid1
    remote_password SharedAccessSignature sr=myiothub.azure-devices.net%2Fdevices%2Fmydeviceid1&sig=aSdsFp%2FbzHjujs6aPOn%2Bo3OeaMLCuD8HGd9KJDksZc%3D&se=1496852317
    clientid mydeviceid1
    bridge_cafile /etc/ssl/certs/ca-bundle.crt
    try_private false
    cleansession true
    start_type automatic
    bridge_insecure false
    bridge_protocol_version mqttv311
    bridge_tls_version tlsv1
    notifications false
    notification_topic events/
    topic # out 0 devices/mydeviceid1/messages/events/ devices/mydeviceid1/messages/events/
    topic # in 0 devices/mydeviceid1/messages/devicebound/ devices/mydeviceid1/messages/devicebound/
connection iothub-bridge2
    log_type debug
    address myiothub.azure-devices.net:8883
    remote_username myiothub.azure-devices.net/mydeviceid2
    remote_password SharedAccessSignature sr=myiothub.azure-devices.net%2Fdevices%2Fmydeviceid2&sig=aSdsFp%2FbzHjujs6aPOn%2Bo3OeaMLCuD8HGd9KJDksZc%3D&se=1496852317
    clientid mydeviceid2
    bridge_cafile /etc/ssl/certs/ca-bundle.crt
    try_private false
    cleansession true
    start_type automatic
    bridge_insecure false
    bridge_protocol_version mqttv311
    bridge_tls_version tlsv1
    notifications false
    notification_topic events/
    topic # out 0 devices/mydeviceid2/messages/events/ devices/mydeviceid2/messages/events/
    topic # in 0 devices/mydeviceid2/messages/devicebound/ devices/mydeviceid2/messages/devicebound/

Note1: Add file /etc/mosquitto/conf.d/bridge.conf

Note2: Install certfiles 

sudo vim /etc/mosquitto/conf.d/bridge.conf
    sudo wget http://curl.haxx.se/ca/ca-bundle.crt
    sudo cp ca-bundle.crt /etc/ssl/certs/

Note3: Generate the SharedAccessSignature from the device explorer

Note4: Add multiple sets of the configurations to the bridge.conf for multiple devices

Note5: Running mosquitto bridge 

sudo mosquitto -c /etc/mosquitto/conf.d/bridge.conf -v
